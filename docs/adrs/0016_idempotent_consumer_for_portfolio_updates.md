# ADR-0016: Idempotent Consumer for Portfolio Updates

**Date**: 2026-03-31

## Status

Accepted

## Context

`portfolio-service` consumes `OrderApprovedEvent` from Kafka and calls `upsertHolding()` to add a holding. Kafka guarantees **at-least-once delivery**: if a consumer processes a message but crashes before committing the offset, the broker re-delivers the same message on restart. Without a deduplication guard, `upsertHolding()` would be applied twice, incorrectly doubling the holding quantity.

This was observed in practice: taking `portfolio-service` offline mid-transaction and bringing it back caused a 2 BTC order to result in 4 BTC in the portfolio.

The event already carries a stable, globally unique `transactionId` (UUID generated by `transaction-service` at order placement). This is the natural idempotency key.

## Decision

Introduce a `processed_transaction` table in `portfolio-service`'s database. At the start of `upsertHolding()`, insert the `transactionId` with a `UNIQUE` constraint. If the insert raises a `DataIntegrityViolationException`, the event has already been processed — log a warning and return immediately. If the insert succeeds, proceed with the holding update. Both the insert and the holding update execute within the same `@Transactional` boundary, so they commit atomically or roll back together.

```
Consumer receives OrderApprovedEvent
  └─ upsertHolding(transactionId, userId, symbol, amount, price)
       ├─ INSERT INTO processed_transaction (transactionId) → success → apply holding update → commit
       └─ INSERT INTO processed_transaction (transactionId) → DataIntegrityViolationException → skip, return
```

The DB constraint is the authoritative guard. No pre-check (`existsById`) is needed — relying on the constraint avoids the check-then-act race condition and keeps the logic simple.

## Consequences

**Positive**
- Re-delivered messages are silently discarded; holding quantities remain correct after any number of consumer restarts.
- No distributed coordination needed — the deduplication is entirely within `portfolio-service`'s own ACID transaction.
- `processed_transaction` doubles as an audit log: every applied `OrderApprovedEvent` is permanently recorded with its `processedAt` timestamp.
- Works correctly under partition rebalancing, rolling restarts, and crash recovery scenarios.

**Negative**
- Each `upsertHolding()` call incurs an additional DB write. Acceptable for the current order volume.
- `processed_transaction` grows unboundedly. For production use, a periodic cleanup job or retention policy for old rows should be added.

## Alternatives Considered

**Pre-check with `existsByTransactionId()`**
Read before write: check if the `transactionId` is already in the table, skip if present.

- Rejected: introduces a time-of-check/time-of-use race condition. Two concurrent consumer threads (possible during partition rebalancing) could both pass the check before either commits the insert. The constraint-based approach is race-free by relying on the DB for mutual exclusion.

**Exactly-once semantics (Kafka transactions)**
Configure Kafka producer + consumer with `isolation.level=read_committed` and transactional producers.

- Rejected: requires the Kafka producer (in `transaction-service`) and consumer (in `portfolio-service`) to both participate in Kafka transactions. This adds significant configuration complexity and ties two independent services into a distributed transaction, violating the data-ownership boundary (ADR-0007). The DB-level guard achieves the same result within a single service boundary.
